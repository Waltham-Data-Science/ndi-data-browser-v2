## Executive summary

The codebase has **solid fundamentals**. The proxy architecture is well-thought-through: Fernet-encrypted session storage, CSRF double-submit, per-user cache scoping, a URL-bearer-forwarding allowlist, path-traversal-proof static serving, HTTP/2 pool with circuit breaker, Prometheus histograms, structlog, pydantic validators on most routes, 88% backend test coverage (live-measured, not the 5.75% a stale committed `coverage.xml` claims), 395 tests passing. Frontend bundle is already tree-shaken well (360 KB gzipped total, SheetJS correctly lazy-loaded).

But four issues are **load-bearing and exploitable today** and should be treated as P0:

1. **üö® CRITICAL ‚Äî SSRF in the binary-download client.** `NdiCloudClient.download_file(url, ...)` fetches any absolute URL the cloud hands it, with zero scheme/host validation on the GET itself (the existing allowlist only gates Authorization-header forwarding, and only when `DOWNLOAD_ALLOWLIST_ENFORCE=True`, which is still off in phase-1). A dataset author ‚Äî or a compromised cloud response ‚Äî can set a document's `files.file_info.locations.location` to `http://169.254.169.254/latest/meta-data/iam/security-credentials/` (AWS metadata) or `http://localhost:6379/` (Redis in the Railway container) and receive the bytes back as an image/timeseries stream. Works **unauthenticated on public datasets**.

2. **üö® CRITICAL ‚Äî Cross-user data leak via cookie-name drift.** `CacheControlMiddleware` looks for a cookie named `ndi_session=` but the session cookie is actually `session=` (per `SESSION_COOKIE` in `auth/dependencies.py:17`, confirmed in `_deps.py:144` and in every integration test). The check never matches, so `_cache_control_for` always returns `public, max-age=60, s-maxage=60` ‚Äî even on authenticated responses like `/api/datasets/my` and `/api/auth/me`. No `Vary: Cookie` is set either. Any shared cache in front of Railway (Cloudflare, corporate MITM proxy, future CDN) will cache the first user's response and serve it to subsequent users for 60 s.

3. **üö® CRITICAL ‚Äî No route-level code splitting on the frontend.** Every route is a static import in `App.tsx`; zero `React.lazy` / `<Suspense>` anywhere in the codebase. The main 298 KB / 86.6 KB gzipped bundle ships LoginPage, QueryPage, MyDatasetsPage, DocumentDetailPage, DataPanel (which transitively pulls uPlot into the main graph), AboutPage, DependencyGraph to every first-time visitor of `/datasets`. ~40‚Äì60 KB gzipped of unnecessary work for first paint on the most-trafficked entry.

Beyond those three: a **HIGH local-hygiene finding** ‚Äî two macOS Finder duplicate files (`backend/middleware/cache_control 2.py`, `backend/routers/_validators 2.py`) exist in the working tree but are **untracked in git** and thus never shipped to Railway. They came from an iCloud Drive / Finder dedup. They don't affect prod today, but they're one `git add -A` away from it and they distort local coverage runs ‚Äî three of the four audit agents saw them on disk and assumed they were in the image. Delete locally + add a CI hygiene check so they can't sneak in later.

Also 22 HIGH findings including a classic login-CSRF (login exempt from CSRF with no backend Origin check), docs drift on the superseded Cognito refresh flow (ADR-008 removed it 2026-04-17; workflows/architecture/README still describe it as current), concurrency throttles misaligned between DatasetSummaryService (=3) and SummaryTableService (=6), `.dockerignore` doesn't exclude `backend/tests/` so fixtures ship to prod, visualize router bypasses the shared `DatasetId` validator, frontend coverage not measured or gated, and three E2E specs that the docs cite by name don't exist on disk.

Total findings: **86** (3 CRITICAL, 23 HIGH, 40 MEDIUM, 20 LOW). Known/tracked items from the CLAUDE.md deferred list (`DOWNLOAD_ALLOWLIST_ENFORCE=False`, non-atomic Redis rate-limit pipeline, `unknown` IP fallback, missing CSP `<meta>`, Cognito 1-h TTL, E2E manual-only, flaky `test_ip_change_logs_warning_allows_request`, SheetJS CDN tarball, single Railway replica) are **not** re-filed ‚Äî I verified each is still present and still tracked.

## P0 ‚Äî File within 24 h

### [CRITICAL-1] SSRF: binary-download client fetches any absolute URL

**File:** `backend/clients/ndi_cloud.py:484-528`

**Issue.** `download_file(url, *, access_token)` takes an absolute URL from the upstream document payload and calls `self.client.get(url, headers=headers, timeout=60.0)` with zero scheme/host validation on the GET itself. The existing allowlist (`_url_allowlist.py`) is wired in (`extract_host` + `host_matches_allowlist`), but all it does is decide whether to forward the Bearer token ‚Äî and even that gating is phase-1-off (`DOWNLOAD_ALLOWLIST_ENFORCE=False` by default, tracked). The request itself always fires.

**Impact.** A dataset author ‚Äî or a compromised cloud response, or a MITM on an S3 signed URL ‚Äî can set a document's `files.file_info.locations.location` to:
- `http://169.254.169.254/latest/meta-data/iam/security-credentials/` ‚Üí AWS container credentials
- `http://localhost:6379/INFO` ‚Üí Redis INFO dump
- `file:///etc/passwd` (httpx does allow `file://` by default)
- `http://redis.railway.internal:6379/` ‚Üí Railway's internal Redis via the private network

Then call `GET /api/datasets/:id/documents/:docId/data/image` (or `/timeseries`, or `/fitcurve`). Works on **public datasets with no auth**.

**Fix.**

```python
from urllib.parse import urlparse

async def download_file(self, url: str, *, access_token: str | None = None) -> bytes:
    parsed = urlparse(url)
    if parsed.scheme not in {"http", "https"}:
        raise BinaryNotFound()  # or a dedicated UntrustedLocation error
    host = parsed.hostname or ""
    allowlist = self._runtime_download_allowlist()
    if not host_matches_allowlist(host, allowlist):
        log.warning("cloud.download.off_allowlist_host", host=host, ...)
        raise BinaryNotFound()  # hard-reject, don't soft-forward
    # ... rest unchanged
```

This is a single 8-line change. The soft-observe phase is already over; the observation period's purpose was confirming no legitimate traffic went off-allowlist, and the logs can be checked quickly. Flip hard-reject today.

### [CRITICAL-2] Cross-user data leak: CacheControlMiddleware stamps `public, s-maxage=60` on authenticated responses

**File:** `backend/middleware/cache_control.py:256-264` vs `backend/auth/dependencies.py:17`

**Issue.** `_has_session_cookie(scope)` checks for a cookie starting with `ndi_session=`. But the actual session cookie is named `session` ‚Äî set by `do_login` via `SESSION_COOKIE = "session"` in `auth/dependencies.py:17`, read the same way in `routers/_deps.py:144` and in every integration test (`tests/integration/test_routes.py:155` et al.) as `client.cookies.set("session", ...)`. The middleware's comment even documents the wrong name, making this a textbook code/comment/implementation triple-drift.

The logic: `is_authed = _has_session_cookie(scope)` ‚Üí always `False` ‚Üí response gets `f"public, max-age={max_age}, s-maxage={max_age}"` unconditionally. No `Vary: Cookie` either.

**Impact.** Today there is no Cloudflare or shared CDN in front of `ndb-v2-production.up.railway.app`, so the bug is latent. But:
- The moment anyone puts a CDN in front (common hardening step), every authenticated GET starts leaking: `/api/datasets/my`, `/api/auth/me`, dataset-detail for in-review datasets, etc.
- Corporate-proxy MITM is already a concern for HIPAA-adjacent deployments ‚Äî some enterprise networks serve their own cache layer transparently.
- The design intent was explicitly "CDNs must not share authenticated responses" (read the comment in the file). The implementation silently violates that.

**Fix.** Two-line change (+ add `Vary` unconditionally as defense in depth):

```python
# backend/middleware/cache_control.py
def _has_session_cookie(scope: Scope) -> bool:
    cookie = _header_get(scope, b"cookie") or ""
    return any(
        part.strip().startswith("session=") for part in cookie.split(";")
    )

def _cache_control_for(scope: Scope, ...) -> str:
    # ... unchanged body ...
    # ADD: always set Vary: Cookie, Accept-Encoding (via the enclosing middleware,
    # regardless of whether is_authed ‚Äî defense in depth against cookie-name drift)
```

Also add an integration test: hit an authed route with a valid session, assert `Cache-Control` header contains `private` and NOT `s-maxage`.

### [CRITICAL-3] No route-level code splitting ‚Äî entire app ships on first paint

**File:** `frontend/src/App.tsx:10-25`

**Issue.** Every page is a static `import` at the top of `App.tsx`. Zero `React.lazy` or `<Suspense>` anywhere in the repo (`grep -rE "React\.lazy|lazy\(" frontend/src` returns nothing). The main `index-*.js` bundle is 298 KB raw / 86.6 KB gzipped ‚Äî ~40‚Äì60 KB gzipped of which is routes a public visitor to `/datasets` never opens (LoginPage, QueryPage, MyDatasetsPage, DocumentDetailPage, DataPanel with its uPlot trigger, AboutPage, DependencyGraph).

uPlot is already in its own chunk (correct!), but because DataPanel is statically imported from DocumentDetailPage (which is statically imported from the router), the uPlot chunk still downloads on every `/datasets` visit.

**Impact.** TTI and LCP both suffer on the highest-traffic entry path. Mid-range laptops on 4G see ~300‚Äì500 ms extra on first paint.

**Fix.** 6‚Äì8 `React.lazy` conversions + one `<Suspense>` wrapper:

```tsx
const LoginPage = lazy(() => import('@/pages/LoginPage'));
const QueryPage = lazy(() => import('@/pages/QueryPage'));
const MyDatasetsPage = lazy(() => import('@/pages/MyDatasetsPage'));
const DocumentDetailPage = lazy(() => import('@/pages/DocumentDetailPage'));
const DocumentExplorerPage = lazy(() => import('@/pages/DocumentExplorerPage'));
const AboutPage = lazy(() => import('@/pages/AboutPage'));

<Routes>
  <Route element={<Suspense fallback={<PageSkeleton />}><Outlet /></Suspense>}>
    <Route path="/login" element={<LoginPage />} />
    {/* ... */}
  </Route>
</Routes>
```

The skeleton should match the hero band geometry so the handoff is invisible. ~2‚Äì3 hours of work.

## HIGH ‚Äî 22 findings

### Security / correctness (4)

### [HIGH-A1] `/api/auth/login` CSRF-exempt with no backend Origin/Referer check

**File:** `backend/middleware/csrf.py:47-49`

**Issue.** `EXEMPT_PATHS` contains `/api/auth/login`. The frontend already sends `X-XSRF-TOKEN` on login (`frontend/src/api/client.ts:46-49`), but the middleware ignores it for that path, and there's no `Origin`/`Referer` allowlist in the login handler.

**Impact.** Classic login-CSRF: an attacker on `evil.com` can force the victim's browser to POST `/api/auth/login` with the attacker's credentials. The victim's subsequent authenticated actions (browsing, bookmarks, uploads, queries) land in the attacker's account ‚Äî a session-fixation variant that's silent from the user's perspective.

**Fix.** One-line: drop `/api/auth/login` from `EXEMPT_PATHS`. The frontend already sends the token.

### [HIGH-A2] `VisualizeService` accepts unvalidated `datasetId` (path injection)

**File:** `backend/routers/visualize.py:17-21`

**Issue.** `DistributionBody.datasetId` has `min_length=1, max_length=64` but no `pattern`. The value flows into `cloud.get_document_class_counts(dataset_id)` which does `f"/datasets/{dataset_id}/document-class-counts"`. httpx normalizes `..` but preserves other path characters.

**Impact.** `POST /api/visualize/distribution` with `{"datasetId": "foo/bar"}` cleanly hits `/datasets/foo/bar/document-class-counts` upstream. Pivots into sibling resources at minimum; breaks the "proxy is sole validation" contract.

**Fix.** Switch `DistributionBody.datasetId` to use the shared `DatasetId` validator from `_validators.py` (regex `^[a-zA-Z0-9_\-]{1,128}$`). Same change worthwhile for `className`, `field`, `groupBy`.

### [HIGH-A3] `do_logout` doesn't clear cookies on non-network cloud errors

**File:** `backend/auth/login.py:128-144`

**Issue.** `cloud.logout()` swallows `CloudUnreachable`/`CloudTimeout` but propagates anything else (`CloudInternalError`, `Forbidden`, etc.). The `response.delete_cookie` calls live **after** the try/finally ‚Äî if the cloud call raises a non-swallowed exception, the browser keeps both `session` and `XSRF-TOKEN` cookies for 24 h.

**Impact.** Logout UX breaks on upstream 500s. The stuck session cookie points at deleted Redis state, producing 401 loops until expiry. Cosmetic security concern (no confidentiality loss) but real UX failure.

**Fix.** Move `delete_cookie` into the `finally` block; broaden the cloud-logout except to all cloud errors.

### [HIGH-A4] `get_current_session` raises on unexpected Redis payload shapes

**File:** `backend/auth/session.py:186-190`

**Issue.** The `get()` exception handler catches only `CorruptSession` and `InvalidToken`. A `KeyError`/`TypeError`/`ValueError` from `SessionData.from_redis` on a drifted-schema Redis blob propagates to FastAPI's default 500 handler. `CorruptSession` itself is defined but never actually raised ‚Äî dead code.

**Impact.** One bad blob crashes every session read ‚Äî post-deploy schema drift or partial Redis write during eviction triggers a 500 burst instead of a clean fall-through to re-login.

**Fix.** Broaden to `except (CorruptSession, InvalidToken, KeyError, TypeError, ValueError)`, log at WARN, delete the corrupt blob, fall through to anonymous. Drop the unused `CorruptSession` class.

### Backend perf / architecture (6)

### [HIGH-B1] Eager scipy/numpy/PIL imports at app boot

**File:** `backend/routers/visualize.py`, `backend/services/visualize_service.py:12-13`, `backend/services/binary_service.py:29-30`

**Issue.** `app.py` ‚Üí `routers/visualize` ‚Üí `VisualizeService` top-level-imports `from scipy import stats` (355 ms cumulative) and numpy. `binary_service` top-level-imports PIL + numpy. Every worker pays this on every cold start, whether `/visualize` or `/binary` is ever called.

**Impact.** ~500 ms √ó 4 uvicorn workers = ~2 s extra cold start per Railway deploy. Compounded on rolling redeploys.

**Fix.** Lazy-import inside the methods that use them:

```python
# backend/services/visualize_service.py
async def distribution(self, ...):
    from scipy import stats  # noqa: local import
    import numpy as np
    # ...
```

### [HIGH-B2] `pandas>=2.2.0` declared but never imported

**File:** `backend/pyproject.toml:21`, `backend/requirements.txt:14`

**Issue.** `import pandas` is in zero files (`grep -r 'import pandas' backend/` returns nothing). But the wheel (~30 MB) installs in every Docker build.

**Impact.** Image bloat. If anyone imports it accidentally, 6.3 s cold-start cost.

**Fix.** Remove from `pyproject.toml` + `requirements.txt`.

### [HIGH-B3] `OntologyCache` opens a fresh SQLite connection per `get`/`set`

**File:** `backend/services/ontology_cache.py:66-76, 103-109`

**Issue.** `_conn()` is called inside every method ‚Äî opens a new connection, runs `PRAGMA journal_mode=WAL`, returns. Serializes through `threading.Lock`. Worst case: `batch_lookup(25)` = 25 sequential connection-opens-under-lock.

**Impact.** Ontology cache hits on summary builds (the common case) serialize through one-by-one synchronous SQLite opens, blocking the event loop. Measurable during warmup and summary-service fallthrough.

**Fix.** Hold one connection per process in `__init__` with `check_same_thread=False` + keep the existing lock for write serialization. Or adopt `aiosqlite` for async end-to-end.

### [HIGH-B4] `DatasetSummaryService.MAX_CONCURRENT_BULK_FETCH = 3` vs `SummaryTableService = 6`

**File:** `backend/services/dataset_summary_service.py:59` vs `summary_table_service.py:54`

**Issue.** The comment claims alignment, but the values differ. Catalog list pages also layer `MAX_CONCURRENT_SUMMARIES=3` on top of the 3-way per-summary fan-out, giving effectively 9-way concurrent bulk-fetches for a 100-row catalog ‚Äî far below what the summary-table path can push.

**Impact.** A 100-row catalog page with 50 cache-miss summaries takes ‚âà17 serialized summary rounds.

**Fix.** Raise `MAX_CONCURRENT_BULK_FETCH` to 6. Benchmark `MAX_CONCURRENT_SUMMARIES=6` too (watch Lambda concurrency).

### [HIGH-B5] `FacetService._collect_full_summaries` walks the full catalog on cold cache

**File:** `backend/services/facet_service.py:226-257`

**Issue.** Facet build walks up to 5000 datasets and calls `summary_service.build_summary()` for each. Today's scale (‚âà100 datasets) √ó summary build = hundreds of cloud calls on cold cache. Every 5 minutes the facet cache expires and the first user pays this.

**Impact.** One cold-cache facets request ‚âà 100 √ó 3 = 300 cloud calls. Lambda cold-start thrash risk at scale.

**Fix.** (a) Short-circuit summary call if `row["species"]` / `brainRegions` / `numberOfSubjects` are present from cloud (the ndi-cloud-node#15 dormant path); (b) pre-warm facet cache on `_keep_warm` cycles; (c) use `DatasetSummary.compact` fields only where possible.

### [HIGH-B6] No `request.is_disconnected()` on long-running routes

**File:** `backend/routers/tables.py`, `datasets.py`, services for summary/pivot/provenance/combined

**Issue.** `/api/datasets/:id/summary`, `/tables/combined`, `/datasets/:id/provenance`, `/datasets/:id/pivot/:grain` can take >10 s on cache miss. Zero `await request.is_disconnected()` checks anywhere in the codebase. The backend keeps pulling bulk-fetches after the client has hung up.

**Impact.** Wasted Lambda invocations on cancelled navigations. On provenance builds with `_MAX_UNIQUE_TARGETS=1000`, a cancelled request still exhausts the full resolution budget.

**Fix.** Insert `await request.is_disconnected()` checks between bulk-fetch rounds. Longer-term: `anyio.create_task_group` + cancellation scope so client disconnect unwinds in-flight cloud requests.

### [HIGH-D5] Frontend coverage is not measured or gated

**File:** `frontend/vitest.config.ts`, `frontend/package.json`

**Issue.** No `coverage` block in vitest config. No `@vitest/coverage-v8` in devDependencies. 18 `*.test.ts[x]` files run in CI via `npm test` but there is no floor ‚Äî someone can delete them all and CI passes.

### [HIGH-D6] `.dockerignore` doesn't exclude `backend/tests/`

**File:** `.dockerignore`

**Issue.** `.dockerignore` excludes `tests-e2e` (frontend) but no rule for `backend/tests/`. `Dockerfile:30` is `COPY backend/ ./backend/`, so 32+ unit test files, integration suite, contract suite, fixtures all ship in the prod image.

**Fix.** Add `backend/tests/` and `backend/**/tests/` to `.dockerignore`.

## MEDIUM ‚Äî 40 findings (summarized)

### Security / correctness

- **M1** Query validator `_walk` recursion has no depth limit (`backend/services/query_service.py:120-128`) ‚Äî nested `or` DoS ‚Üí 500.
- **M2** `_fallback` dict in RateLimiter + `_locks` dicts in RedisTableCache/AsyncTTLCache grow unbounded ‚Äî slow memory leak under Redis outage.
- **M3** `OntologyPopover` renders raw `data.url` (from external OLS) without `safeHref` ‚Äî `javascript:` URL click-jack risk.
- **M4** `/metrics` unauthenticated ‚Äî full Prometheus output public; endpoint labels, login-attempt counts, circuit-breaker state = reconnaissance gold.
- **M5** `/api/documents/list` has no `le=` upper bound on `page` param (inconsistent with `/api/datasets/published`).
- **M6** `/api/auth/csrf` hardcodes `secure=True` inconsistent with login's `secure = settings.ENVIRONMENT != "development"` ‚Äî dev friction.

### Backend perf

- **M7** Cloud-client `request_id` kwarg exists but zero call sites pass it ‚Äî `X-Request-ID` header never sent ‚Üí cloud‚Üîproxy log correlation lost.
- **M8** `CacheControlMiddleware` buffers every JSON response in memory to compute ETag ‚Äî 2-4 MB combined-table responses hit 2√ó memory overhead.
- **M9** Missing `Vary: Cookie, Accept-Encoding` on cacheable responses (defense-in-depth for CRITICAL-2).
- **M10** Per-worker `keepwarm` + ontology warmup = 4√ó the cloud load on 4-worker deploys ‚Äî use Redis `SET NX EX` leader election.
- **M11** `ProxyCaches` in-process TTLCaches aren't shared across workers ‚Äî per-worker staleness gaps up to 60 s.
- **M12** `OntologyService._http = httpx.AsyncClient(...)` unconfigured ‚Äî no HTTP/2, no pool limits, inconsistent with cloud client's 50-pool config.
- **M13** `OntologyService.batch_lookup` unbounded `asyncio.gather` ‚Äî 100+ parallel requests to OLS/SciCrunch on a single summary build = rate-limit risk.
- **M14** `_extract_om_terms` doesn't normalize ontology IDs before dedupe (`ncbitaxon:10090` vs `NCBITaxon:10090` yield duplicate lookups); `_extract_probe_location_terms` does normalize.
- **M15** `DatasetService.detail/class_counts` cache has no negative-result pruning ‚Äî empty `{}` response gets pinned for 60 s.
- **M16** `_aggregate_cross_dataset_edges` walks classes sequentially ‚Äî provenance build latency scales √ó class-count.
- **M17** `_downstream_of` uses `page_size=200` (arbitrary vs `BULK_FETCH_MAX=500`) ‚Äî BFS systematically biased on dense graphs.
- **M18** `app.state.ontology_warmup_task` never cancelled on shutdown ‚Äî noisy shutdown logs.

### Frontend perf / a11y

- **M19** 55 KB uncompressed CSS (10.8 KB gzipped) with ~25% dead from `dark:*` ‚Äî see HIGH-C6.
- **M20** Unmemoized inline `onClick` closures on MyDatasetsPage filter chips + DatasetCard not `React.memo`'d ‚Äî 50-card grid re-render per chip toggle.
- **M21** `VirtualizedTable` inner `TableCell`/`EpochTimeCell` not memoized ‚Äî keystroke in global filter = 200-cell reconciliation.
- **M22** QueryBuilder re-maps operation dropdown on every keystroke ‚Äî 26 `<option>` renders per condition per key.
- **M23** `useFacets` `staleTime: 30_000` vs backend `TTL=300_000` ‚Äî under-utilizes server cache; raise to match.
- **M24** Modal focus-restoration ordering relies on useEffect cleanup; backdrop-click path has fragile ordering.
- **M25** `DocumentExplorer` `<tr onClick>` is a keyboard trap ‚Äî full row is mouse-only interactive.
- **M26** `DatasetsPage` client-side filter rebuilds haystack strings √ó 400 per keystroke.
- **M27** Hero-band pattern SVG duplicated across 7 page modules ‚Äî factor to `<PageHero>` component.

### Docs / tests / infra

- **M28** `docs/error-catalog.md` cites `test_error_catalog.py` ‚Äî actual file is `test_errors.py`.
- **M29** ADR-005 has two contradictory `**Status:**` lines (both "Superseded" and "Accepted").
- **M30** `visualize_service.py` at 18% coverage ‚Äî untested distribution math.
- **M31** `dependency_graph_service.py` at 36% coverage ‚Äî 180-line graph-build block has zero tests.
